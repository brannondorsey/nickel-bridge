# Rule-based SAYC bidding (Option B) — design & rationale

**Status: shelved for reference (2026-07).** We shipped Option A instead — the
SAYC-constrained argmax in `packages/ai/src/bidder.ts` (`constrainedBest`). This doc
records the full design for a rule-based bidder, why Option A may eventually fall
short, and what Option B would risk, so the decision can be revisited with the
context intact.

## Background

Robot bidding is a neural policy (4×1024 MLP, arXiv:2406.10306) trained by
imitating a dataset that was itself generated by WBridge5 *emulating* SAYC (the
OpenSpiel/Lockhart et al. set — an approximation of an approximation). Nothing in
that pipeline enforces a convention: measured over 400 deals, the raw model's
argmax made a bid that broke the machine-checkable requirements of its own SAYC
meaning 11.3% of the time it bid (5-card weak twos, 11-HCP openings, off-shape
jump shifts). For an app whose purpose is *teaching* SAYC — where the UI explains
every bid with pamphlet semantics and partner's bids are the signals the learner
is training on — that's a correctness bug in the product's premise, not a
strength tradeoff.

## Option A (shipped): SAYC-constrained argmax

`Bidder.constrainedBest` takes the model's legal-action distribution and excludes
every **bid** whose exact SAYC meaning carries machine-checkable hand
requirements (`BidMeaning.req`) that the acting hand fails (`saycViolation` in
`packages/core/src/advisor.ts`). Argmax runs over what's left. Design points:

- **Pass is always admissible.** Forbidding an over-promising bid always leaves a
  safe call; forbidding pass *forces* an action, and when the model's preferred
  actions are all excluded, the argmax over the remainder is noise. (Observed in
  development: with pass forbidden after partner's preempt via the 0–5 response
  pass rule, the constrained argmax produced a lunatic 6♦ raise. The asymmetric
  rule fixes the failure class, not just the instance.)
- Calls the explainer can't check — artificial conventions (Stayman, transfers,
  Blackwood, 2♦ waiting…), doubles, and auctions beyond the pamphlet — remain
  under the model's judgment. Conventions demonstrably survive (unit-tested in
  `packages/ai/test/bidder.test.ts`; a 400-deal sweep shows the full menagerie in
  live use).
- Grading (`evaluate`) uses the same constrained choice for `bestCall`, so the
  "robot preferred X" teaching toast can never name a violating bid.

Result: measurable violations went 11.3% → 0.0% with a diff of ~30 lines.

## Where A falls short (and B wouldn't)

1. **A can only veto what the tables can check.** 43% of robot bids in the sweep
   have no exact pamphlet meaning at all, and `req` deliberately skips artificial
   conventions and doubles. In all that space the model still free-styles. B
   derives every bid *from* the tables, so coverage of the tables is coverage of
   the bidding.
2. **A vetoes but doesn't prefer.** When a SAYC-prescribed call exists but the
   model likes an unchecked call better, A lets it (sweep: 56/1566 bids, e.g. a
   generic 5♦ opening where SAYC says pass or 2♦). B would play the prescribed
   call.
3. **A can't express obligations.** SAYC sometimes *requires* action (respond
   with 6+, answer Blackwood, don't pass a forcing bid). A's pass-always-safe
   rule deliberately gives up on enforcing these; B encodes forcing sequences
   natively.
4. **A's floor is the model's competence.** Where the model is simply wrong but
   not checkably wrong (misjudged invites, wrong strain preference), A inherits
   it. B's mistakes are legible rule gaps you can fix with a table row.
5. **Extensions in A are indirect.** Teaching the robots a new convention under A
   means retraining a neural net (practically: impossible here). Under B a new
   convention is a new rule row + spec-test row — the workflow invariant 3
   already establishes for the *explainer*.

## Option B design sketch

Build the bidder from the machinery `packages/core` already has (`sayc.ts` is
~1,000 lines of table-driven meanings, many with `req`; `advisor.ts` checks
them):

1. **Candidate generation.** For the seat to act, enumerate legal calls and keep
   those whose meaning is exact and whose `req` (extended — see risks) the hand
   satisfies. This is `explainBid` run in reverse: from hand → qualifying calls,
   instead of call → meaning.
2. **Priority ladder.** SAYC is essentially a decision list; encode the pamphlet
   ordering as an explicit priority on rule families (e.g. respond to Blackwood ≫
   accept a transfer ≫ answer Stayman ≫ raise partner's major with support ≫ new
   suit up the line ≫ notrump ladder by range ≫ pass). Deterministic tie-break:
   lowest call index. This is the same architecture as saycbridge's rule engines
   (BSD-3 — crib the priority taxonomy, not the code) and BBA/EPBot.
3. **Constraint completion.** `req` today is authored for grading (HCP + suit
   lengths + balanced). B needs it on the artificial families too (Stayman needs
   a 4-card major and invite+ points; a transfer needs the 5-card suit;
   Blackwood responses need the actual ace count — i.e. `req` grows fields like
   `aces`, `stoppers`, or per-rule predicate functions).
4. **Fallback.** When no rule fires (deep competitive auctions), fall back to
   (a) the constrained model — hybrid mode, pragmatic; or (b) natural rules
   (pass unless a clear bid exists) — pure mode, fully explainable. Ship (a),
   migrate toward (b) as coverage grows.
5. **Single source of truth.** The bidder and the explainer read the same tables,
   so the UI explanation of a robot bid is *definitionally* what the robot meant
   — the core pedagogical guarantee A only approximates.

## Option B risks (why we didn't start here)

- **Authoring surface.** The pamphlet's *meanings* fit in 1,000 lines; a
  *decision procedure* must answer every position, including the competitive
  long tail (balancing, sacrifice judgment, lead-directing doubles) where SAYC
  the document is silent. This is weeks of expert authoring plus adjudicating
  what "SAYC + our extensions" even prescribes where the pamphlet is ambiguous.
  saycbridge — a decade of exactly this — still logs "bidder failed to make a
  call" holes.
- **Playing strength.** The NN beats WBridge5-class opposition; a first-cut rule
  bidder will be materially weaker (thin values judgment, no inference from the
  opponents' auction). Fine for teaching, worse for the "beat the robots"
  scoreboard feel; matchpoint results still work since all humans face the same
  robots.
- **A new bug class.** Rule engines fail by mis-firing rules (wrong priority,
  overlapping conditions) — bugs the drift-guard fixture only catches after the
  fact. Every rule needs spec rows; the suite grows accordingly.
- **Full re-pin churn.** Any bidding change invalidates the robot-trace fixture,
  pinned-seed tests, and demo scenario recipes (CLAUDE.md invariant 1). One-time
  cost per rollout, but B's iterative authoring means paying it repeatedly while
  the ladder stabilizes (mitigation: land B behind `AI_MODEL=rules` and only
  re-pin when it becomes the default).
- **Determinism discipline.** All rules must be pure functions of
  (hand, auction); no randomness, no "style" knobs, or duplicate comparability
  breaks (invariant 1).

## Revisit triggers

Reconsider B (or extend A toward it) if any of these bite:

- Learners report the robots making non-SAYC calls A can't see (the 43%
  unchecked space) frequently enough to erode trust in the explanations.
- We want robots that *execute* taught conventions on demand (lesson mode,
  "robot always answers Stayman honestly") — that's B's native ground.
- We want user-selectable conventions/extensions per table — pure model can't;
  rules can gate rule families per settings.
